## 场景：AI Builders Digest 日报生成 + GitHub Pages 部署

**日期**：2026-06-15  
**来源任务**：Automation — AI Builders Digest 日报  
**标签**：`follow-builders` `html-generation` `github-pages` `bilingual-digest`

---

### ✅ 成功打法

**步骤概要**：
1. 调用 `follow-builders` skill 的 `prepare-digest.js` 获取 JSON 内容
2. 用 Node.js 脚本生成双语 HTML（参考前一天的 HTML 样式）
3. 复制 HTML 到 `ai-digest-site` 仓库对应日期目录
4. 更新 `index.html` 的 `ISSUES` 对象（追加新日期条目）
5. `git add` + `git commit` + `git push`
6. 用 `curl` 验证部署成功（HTTP 200）

**关键细节**：
- JSON 输出可能很大，用 `head -c 50000` 会截断；应保存到文件再读取
- HTML 样式严格参考前一天文件（`#faf9f6` 背景，`accent #c2410c`，Inter + Noto Serif SC + JetBrains Mono）
- `index.html` 的 `ISSUES` 对象按日期倒序排列，新条目插到最前面
- 部署验证用 `curl -s -o /dev/null -w "%{http_code}" <url>` 检查 HTTP 200

**踩坑记录**：
- `prepare-digest.js` 的 JSON 输出含完整推文内容和 Podcast 文字稿，需合理截断或精简后放入 HTML
- Bash 命令写入 HTML 时需注意 heredoc 中的变量插值和引号转义
- 如果 shell 没有写入 Desktop 的权限，用 `mkdir -p` + `cp` 可能失败；需检查 sandbox 权限

---

### ✅ 成功打法

**调用方式**：
```bash
cd /Users/workingchloe/.workbuddy/skills/follow-builders/scripts
node prepare-digest.js > /tmp/fb-digest-YYYY-MM-DD.json 2>&1
```

**输出结构**：
- `status`: "ok"
- `config`: 用户偏好（language, frequency, delivery）
- `podcasts`: [{ name, title, url, publishedAt, transcript }]
- `x`: [{ name, bio, tweets: [{ text, url, createdAt }] }]
- `stats`: { podcastEpisodes, xBuilders, totalTweets, blogPosts }

**注意事项**：
- `language` 决定输出语言：`en` / `zh` / `bilingual`
- 如果 `stats.podcastEpisodes` 和 `stats.xBuilders` 都为 0，说明当天没有新内容
- 推文 URL 必须从 JSON 的 `url` 字段获取，不能自己构造
- Podcast 的 `url` 是频道页，具体视频 URL 需在 transcript 中或在 skill 输出中另行获取

## 场景：follow-builders 中央 feed 冻结 → 连续日内容重复

**日期**：2026-07-03  
**来源任务**：Automation — AI Builders Digest 日报  
**标签**：`follow-builders` `feed-freshness` `dedup` `data-quality`

---

### ⚠️ 观察（踩坑预警）

- `prepare-digest.js` 从中央 feed 拉取内容，feed 含 `feedGeneratedAt` 字段。
- 若连续两日 `feedGeneratedAt` 不变 + 全部 tweet `createdAt` 停留在上一天，则当日 digest 与昨日**完全相同**（feed 未刷新）。
- 实测 2026-07-03 的 `feedGeneratedAt = 2026-07-02T07:27Z`，tweets 均在 7/1-7/2、Podcast 7/1，与 7/2 同源。
- **处置**：仍按既定流水线生成并部署（stats 检查通过），但应（1）在自动化记忆中如实标注重复；（2）将该日作为"同一快照的更完整版本"处理——补齐前一日漏渲染的 builder（7/3 渲染 17 位、7/2 仅 12 位）；（3）提示用户决定是否保留该条目，避免污染日历 streak 与"每日覆盖"的可信度。
- **判定规则**：部署前先比对 `feedGeneratedAt` 与前一日；若相同则标记"stale snapshot"，由用户决定是否落库。

## 场景：follow-builders prepare-digest.js 远程 fetch 间歇性失败（000 / "fetch failed"）

**日期**：2026-07-12
**来源任务**：Automation — AI Builders Digest 日报
**标签**：`follow-builders` `fetch-flaky` `retry` `robust-fetch` `network`

### ⚠️ 踩坑

- `prepare-digest.js` 对 3 个 feed + 5 个 prompt 用 `Promise.all([fetchJSON(...), ...])` 一次性拉取，**零重试**。
- 当任一远程 URL（实测 `feed-blogs.json`、`prompts/translate.md`）出现连接级 `fetch failed`（curl 探测返回 `000`，非 404）时，整次 `main()` 被 reject，stderr 仅 `{"status":"error","message":"fetch failed"}`，stdout 为空 → 获取连败 2 次。
- 同会话内 `curl github.com`=200、`node fetch google`=200，说明**不是整体断网**，而是 raw.githubusercontent.com 部分资源间歇性握手失败（DNS/TLS/限流随机掉线）。

### ✅ 打法（已验证）

- 自写健壮抓取脚本（参考 `/tmp/fb-prepare-robust.mjs`），核心：**每个 URL 单独重试**（6 次、指数退避 + 抖动），不共用一个 Promise.all。
- feed 兜底顺序：远程(重试) → 技能目录内置 `feed-*.json`（仅当远程彻底不可达，且**必须打 STALE 标记告知用户，绝不冒充"今日"内容**）→ 否则报错。
- prompt 兜底顺序：用户自定义 > 远程(重试) > 技能内置。
- 输出 schema 与官方 `prepare-digest.js` 完全一致，下游 HTML 生成脚本无需改动。
- **铁律**：技能内置 `feed-*.json` 是 4/27 旧快照，绝不能作为"当天 digest"内容来源；远程失败只能重试，不能拿旧数据顶替。
- 验证：重试脚本一次跑通，103KB 合法 JSON，`feedGeneratedAt` 为当日（2026-07-12T01:19Z）。
